Backtesting Improved Cross Exchange Arb (day-level granularity)¶

See $\texttt{simple}\_\texttt{backtest.ipynb}$ for a description of the prior iteration of this code.

Introduction¶

The process/flow the data goes through is roughly the same as before, the trading logic is what has greatly improved.

Improvements Made:

  1. Trading logic improved to determine strength of indicator vs a threshold
  2. Trading timing improved, not simply daily hold times. Now hold as long as deviation signal exists. This optimizes with respect to trading costs and prfit capture.
  3. Trading costs imposed
  4. More data (> 2 years)

Data Flow:

  1. Pull data from web and save
  2. Pre-compute rolling means and standard deviations of spread between exchange $i$ and $j$ with a 30 day rolling window
  3. Every day compute deviation from mean to determine if the spread movement is strong enough to open new position or weak enough (reverted close enough to mean) to close existing position(s). Apply trading logic shown below
  4. As necessary, apply $1-c$ proprtion of funds to the trade in each "buy" account where $c$ is some cushion $0 < c < 1 $. Liquidate existing positions in "sell" accounts as necessary
  5. Subtract trading costs
  6. Compute account balances and profit over time. Balances cannot go negative, liquidate positions if so.

Trading Logic:

  1. Determine the following for each exchange pair (e.g. Kraken - Bitfinex) $$ \gamma_{ij} = \frac{\text{Spread}^t_{ij} - \text{Spread}^{\mu}_{ij}}{\sigma^\mu_{ij}} $$ where $t$ is the current time, $\mu$ is the rolling window value, $\gamma$ is the number of standard deviations the move in price is, and $ij$ is exhcange $i$ - exchange $j$.
  2. Compute $$ \{ \gamma_{ij}^* \} = \{\gamma_{ij} \space | \space \gamma_{ij} > \sigma_{\text{thresh}} \} $$ and the resulting $\{i^*, j^*\} $ ordered pairs. As a reminder these indices correspond to exchanges and $\sigma_{\text{thresh}}$ is a cutofff value in the range [1.5, 2.5].
  3. To minimize trading costs we want to choose the $n$ highest $\gamma_{ij}^*$'s and trade only these pairs. If there is overlap (e.g. there is signal to trade Kraken-Bitstamp and Kraken-Gemini) we will allocate funds based on the strength of $\gamma_{ij}^*$. $$ \text{%_of_Funds}_{ij}^* = \frac{\gamma_{ij}^*}{\sum\limits_{{k \in \{j^*\}}} \gamma_{ik}^*} $$ where $\text{%_of_Funds}_{ij}^*$ is the percent of funds in account $i^*$ allocated to the $i^*, j^*$ trade. $\{j^*\}$ is the set of all $j^*$ paired with some $i^*$. $n$ will most likely be in the range [1, 3]. Also note that because we never want to be exposed to the underlying movements we restrict $\text{%_of_Funds}_{ij}^* = \text{%_of_Funds}_{ji}^*$. This means that if one account does not have enough funds in it we abondon the $\text{%_of_Funds}_{ij}^*$ approach and simply apply whatever possible to one side of the trade and then the rest to the other side drawing on that account (see example below).
  4. Short currency in $i^*$ exchange and long in $j^*$ exchange in the amount: $(1-c) \times \text{Funds}_{ij}^* \times \text{%_of_Funds}_{ij}^* $.
  5. For all currently held positions determine if $\gamma_{ij}^* \leq \sigma_\text{thresh} $ and if so, liquidate positions. If this frees up capital to move into an existing trade in one of those accounts, apply those funds. For example, if both a Kraken-Gemini and Kraken-Bitfinex trade were open and each had 30 and 70 percent of the Kraken funds respectively. You then determine that the Kraken-Gemini trade is within the threshold. You liquidate that position and apply the proceedds to both the other Kraken position but also the Bitfinex position. This needs to happen so that we remain unexposed to the underlying.

Example:

We compute the proportions in any single acount based on the most frequent short position, remember that the short position is the first index. We also cannot be exposed to the underlying so the trade value is always the minimum account balance of either account partaking in a trade. That is to say if we have the following combinations of $i^*, j^*$, $\{ (2,1), (2,3), (3,4) \}$, with the following $\gamma_{ij}^*$, $\{2.5, 2.1, 3.7 \}$. In English this means that the spread between exchange 2 and exchange 1 has deviated from its mean by 2.1 standard deviations and so on. The proportions in the 2nd account would be $(\frac{2.1}{2.1+2.5}, \frac{2.5}{2.1+2.5}) = (0.54, 0.46)$, which are the proportions we use to compute the balances because the 2nd account is the most frequent short position. Say we have the following balances in accounts 1,2,3,4 respectively: \$1000, \$800, \$950, \$500. We would make the following 4 trades:

  1. Short Account 2 in net amount: \$800(1-c) = \$368(1-c) + \$432(1-c)
    • Short Account 2 in amount: \$800(1-c) x 0.54 = \$432(1-c)
    • Short Account 2 in amount: \$800(1-c) x 0.46 = \$368(1-c)
  2. Long Account 1 in amount: \$368(1-c)
  3. Short Account 3 in net amount: \$68 = \$500(1-c) - \$432(1-c)
    • Long account 3 in amount: \$432(1-c)
    • Short account 3 in amount: \$500(1-c) _This would normally be \$518(1-c) to use the rest of the third accounts value (it is the short position so it takes precedence), but account 4 only has \$500 in it so we cannot go over this._
  4. Long account 4 in amount: \$500(1-c)

To summarize, in order to determine the trade flows the order of precedence is (from most important to least):

  1. Accounts that will be overdrawn if a flow is made
  2. Pairs with higher $\gamma_{ij}^*$
  3. Accounts with the most frequent short flows
  4. Accounts with the least frequent short flows
  5. Accounts with the most frequent long flows
  6. Accounts with the least frequent long flows

    uniq = list(np.unique(lst))
    freq = [lst.count(x) for x in uniq]
    max_f = max(freq)
    return [uniq[i] for i in range(len(uniq)) if freq[i]==max_f]
